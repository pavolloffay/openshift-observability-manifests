## Install

```bash
oc apply -f 00-install-operator.yaml
```

Wait for the operator to be ready, then create the Azure OpenAI secret from your service principal credentials:

```bash
oc create secret generic azure-openai-api-keys -n openshift-lightspeed \
  --from-literal=client_id='<appId>' \
  --from-literal=client_secret='<password>' \
  --from-literal=tenant_id='<tenant>'
```

Or edit `01-configure-api-keys.yaml` with your values and apply it:

```bash
oc apply -f 01-configure-api-keys.yaml
oc apply -f 02-deploy-olsconfig.yaml
```

## Agentic Lightspeed Operator

The agentic operator's `LLMProvider` CRD does not support Azure service principal auth directly.
A [LiteLLM](https://docs.litellm.ai/) proxy is deployed as a bridge — it accepts OpenAI-compatible
requests and forwards them to Azure OpenAI using the service principal credentials.

### Secrets

The following secrets must exist in the `openshift-lightspeed` namespace before deploying:

| Secret | Keys | Purpose |
|--------|------|---------|
| `azure-openai-api-keys` | `client_id`, `client_secret`, `tenant_id` | Azure service principal credentials (shared with OLS operator) |
| `litellm-master-key` | `LITELLM_MASTER_KEY` | Master key for the LiteLLM proxy (included in `03a-deploy-litellm.yaml`) |
| `agentic-llm-credentials` | `OPENAI_API_KEY` | Must match the LiteLLM master key (included in `04-deploy-agenticolsconfig.yaml`) |
| `agentic-operator-webhook-certs` | `tls.crt`, `tls.key` | TLS cert for the webhook server (auto-generated by OpenShift via Service annotation) |

## Agentic Lightspeed

The Proposal is the only user-facing entry point. Everything else is either configuration or operator-managed output:

You create:
- Proposal (namespaced) — the trigger that kicks off the whole workflow

Cluster admin configures:
- Agent (cluster-scoped) — defines available agents (e.g., "default") and their capabilities
- LLMProvider (cluster-scoped) — configures the LLM backend (model, endpoint)
- AgenticOLSConfig (cluster-scoped) — global operator configuration
- ApprovalPolicy (cluster-scoped) — rules for whether proposals need human approval before execution

Operator creates automatically (from a Proposal):
- AnalysisResult (namespaced) — diagnosis + plan
- ProposalApproval (namespaced) — approval gate
- ExecutionResult (namespaced) — execution outcome
- VerificationResult (namespaced) — post-execution verification
- EscalationResult (namespaced) — likely for when the agent can't resolve the issue and escalates

So yes, it all starts with a Proposal. The operator watches for new Proposals and drives them through the lifecycle, creating the result CRs along the way.


┌────────────────────┬────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│      Resource      │                                                                      Purpose                                                                       │
├────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ AnalysisResult     │ The agent's diagnosis and proposed plan — what it found, what actions it recommends, RBAC it needs, risk assessment, rollback plan                 │
├────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ ExecutionResult    │ Outcome of actually running the proposed actions — which commands succeeded/failed, what changed on the cluster                                    │
├────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ VerificationResult │ Post-execution check — did the fixes actually work? Runs the verification steps defined in the analysis (e.g., "pods healthy", "exporter success") │
├────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ ProposalApproval   │ Approval gate — whether a human approved the proposal before execution proceeds                                                                    │
└────────────────────┴────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

They represent stages in the proposal lifecycle: Analysis → Approval → Execution → Verification.

## How Execution RBAC Works

The operator dynamically provisions RBAC for each proposal's execution phase based on the `rbac` section in the `AnalysisResult`. This is not configured in manifests — it's generated by the LLM during analysis.

### Security Model

| Component | ServiceAccount | RBAC | Purpose |
|-----------|----------------|------|---------|
| Agentic operator | `lightspeed-agentic-operator` | `cluster-admin` | Must create any RBAC the LLM requests |
| Sandbox pod (per-proposal) | `ls-exec-<namespace>-<proposal>` | Only what LLM justified | Minimal, scoped permissions |

The `ls-exec-*` prefix stands for "LightSpeed execution". Each proposal gets its own ServiceAccount with only the permissions the LLM requested in the analysis — not cluster-admin.

### RBAC Flow

1. **Analysis phase**: The LLM analyzes the request and produces an `AnalysisResult` with:
   - Diagnosis and proposed actions
   - Required RBAC permissions (namespace-scoped and/or cluster-scoped)
   - Each RBAC rule must include a `justification` field explaining why the permission is needed

2. **Execution phase**: The operator:
   - Creates a per-proposal ServiceAccount: `ls-exec-<namespace>-<proposal-name>`
   - Creates Role/RoleBinding (for namespace-scoped rules) in each target namespace
   - Creates ClusterRole/ClusterRoleBinding (for cluster-scoped rules)
   - Launches the sandbox pod with that ServiceAccount

3. **Cleanup**: After execution completes, the operator deletes all RBAC resources it created.

### Human Review of RBAC

To require manual approval before execution (so you can review the RBAC the LLM requested):

```yaml
apiVersion: agentic.openshift.io/v1alpha1
kind: ApprovalPolicy
metadata:
  name: cluster
spec:
  stages:
    - name: Analysis
      approval: Automatic
    - name: Execution
      approval: Manual  # Review AnalysisResult RBAC before approving
```

### Example AnalysisResult RBAC Section

```yaml
# This is OUTPUT from the LLM, not user configuration
status:
  options:
  - rbac:
      namespaceScoped:
      - apiGroups: [""]
        resources: ["pods"]
        verbs: ["get", "list"]
        namespace: openshift-lightspeed
        justification: "List pods to check health status"  # Required field
      - apiGroups: [""]
        resources: ["pods/log"]
        verbs: ["get"]
        namespace: openshift-lightspeed
        justification: "Read logs from failing containers"
      clusterScoped:
      - apiGroups: [""]
        resources: ["nodes"]
        verbs: ["get", "list"]
        justification: "Check node status for scheduling issues"
```

### Troubleshooting

If execution fails with "cluster access commands are not available":

1. Check if RBAC was created: `oc get roles,rolebindings -n <target-namespace> | grep ls-exec`
2. Check the AnalysisResult for valid RBAC: `oc get analysisresults -n openshift-lightspeed -o yaml`
3. Verify the `justification` field is present in each RBAC rule — the LLM sometimes omits it, causing validation failures
